# Italian Bible Study вЂ” PWA

A Progressive Web App for a 37-week Italian language study programme using La Bibbia CEI 2008 as the primary text, alongside Babbel, Anki, and iTalki.

Three tabs: **Tracker**, **Flashcards**, and **Journal**. Installs to home screen. Works fully offline.

## Features

| Tab | What it does |
|---|---|
| **Tracker** | 37-week curriculum with expandable week panels, vocab tables, grammar notes, writing prompts, daily schedule, iTalki prompts. Progress persists via localStorage. |
| **Flashcards** | Downloads pre-generated Anki `.apkg` decks for every week, each phase, and the complete 259-card set. Decks regenerate automatically at build time. |
| **Journal** | Per-week writing workspace tied to each week's prompt. Auto-saves as you type (debounced, localStorage). Export all entries as a single `.md` file. |

## Local development

### Prerequisites

- Node.js 18 or higher вЂ” [nodejs.org](https://nodejs.org)
- npm 9 or higher (included with Node.js)
- Anki desktop app (optional, to test imported decks) вЂ” [apps.ankiweb.net](https://apps.ankiweb.net)

Check your versions:

```bash
node --version   # must be 18+
npm --version    # must be 9+
```

### Setup

```bash
unzip italian-bible-pwa.zip
cd italian-bible-pwa
npm install
```

### Generate Anki decks (first time or after vocab changes)

The Anki `.apkg` files in `public/anki/` are generated by a Node script. They are included in the zip, but if you change any vocabulary in `scripts/generate-anki.cjs` you need to regenerate them:

```bash
npm run generate-anki
```

This writes 42 files to `public/anki/`: one per week (37), one per phase (4), and one complete deck.

> **Note:** `npm run build` automatically runs `generate-anki` first via the `prebuild` hook. You only need to run it manually when developing the Flashcards tab.

### Start the dev server

```bash
npm run dev
```

App runs at `http://localhost:5173`. Hot module replacement is active вЂ” edits reflect immediately.

> The service worker and PWA install prompt are disabled in dev mode by design. Use `npm run preview` to test those.

### Preview the production build locally

```bash
npm run build
npm run preview
```

Serves the production build at `http://localhost:4173`. The service worker is active here вЂ” test offline mode by toggling the network in DevTools under **Application > Service Workers**.

## Building for production

```bash
npm run build
```

Order of operations:

1. `prebuild` вЂ” runs `generate-anki.cjs`, writes all `.apkg` files to `public/anki/`
2. Vite builds the React app to `dist/`
3. `vite-plugin-pwa` generates `dist/sw.js` and `dist/workbox-*.js`

Output in `dist/`:

| File / folder | Purpose |
|---|---|
| `index.html` | App shell |
| `assets/index-*.js` | Bundled React app |
| `assets/index-*.css` | All styles |
| `sw.js` | Service worker |
| `workbox-*.js` | Workbox caching runtime |
| `manifest.webmanifest` | PWA manifest |
| `anki/` | All 42 `.apkg` files |
| `icon-192.png`, `icon-512.png` | PWA icons |
| `staticwebapp.config.json` | Azure routing config |

Deploy the entire contents of `dist/` to any static host.

## Deploying to Azure Static Web Apps

Azure Static Web Apps (SWA) is the recommended host. The free tier covers this app with no cost.

---

### Option A: Azure Portal + GitHub Actions (recommended)

Azure commits a GitHub Actions workflow to your repo and deploys automatically on every push to `main`.

#### Step 1: Push to GitHub

```bash
cd italian-bible-pwa
git init
git add .
git commit -m "Initial commit"
```

Create a new private repo at [github.com/new](https://github.com/new), then:

```bash
git remote add origin https://github.com/YOUR_USERNAME/YOUR_REPO.git
git branch -M main
git push -u origin main
```

#### Step 2: Create the Static Web App in the Azure Portal

1. Go to [portal.azure.com](https://portal.azure.com) and sign in.
2. Click **Create a resource**, search **Static Web App**, click **Create**.
3. Fill in the form:
   - **Subscription:** yours
   - **Resource group:** create new, e.g. `italian-bible-rg`
   - **Name:** `italian-bible-pwa` (globally unique)
   - **Plan type:** Free
   - **Region:** closest to you
   - **Source:** GitHub
4. Click **Sign in with GitHub**, authorise Azure.
5. Select your **Organisation**, **Repository**, **Branch** (`main`).
6. Under **Build Details**:
   - **Build Presets:** Vite
   - **App location:** `/`
   - **Output location:** `dist`
7. Click **Review + create**, then **Create**.

Azure commits a workflow file to `.github/workflows/` in your repo. The first build starts immediately вЂ” watch it in the **Actions** tab on GitHub.

#### Step 3: Confirm the deployment

When the workflow finishes (2-4 minutes):

1. Open your Static Web App in the Azure Portal.
2. Click the **URL** on the overview page.
3. The app is live at `https://YOUR_APP_NAME.azurestaticapps.net`.

Every `git push` to `main` triggers an automatic redeploy, which includes regenerating the Anki decks via the `prebuild` hook.

#### Step 4: SPA routing and MIME types

The file `public/staticwebapp.config.json` handles this automatically. It is included in the project and copied to `dist/` at build time. No manual action needed вЂ” but here is what it contains for reference:

```json
{
  "navigationFallback": {
    "rewrite": "/index.html",
    "exclude": ["/assets/*", "/*.{js,css,png,svg,ico,webmanifest}", "/anki/*"]
  },
  "globalHeaders": {
    "Cache-Control": "public, max-age=0, must-revalidate"
  },
  "mimeTypes": {
    ".webmanifest": "application/manifest+json",
    ".apkg": "application/octet-stream"
  }
}
```

The `/anki/*` exclusion ensures `.apkg` file download requests are served directly rather than being rewritten to `index.html`.

If you do not yet have this file in your repo, create it at `public/staticwebapp.config.json` with the content above, commit, and push.

The CLI authenticates via browser and prints the deployment URL when done. To redeploy: `npm run build && swa deploy ./dist ...`.

## Custom domain

1. In the Azure Portal, open your Static Web App.
2. Click **Custom domains** in the left menu, then **+ Add**.
3. Enter your domain (e.g. `bibbia.yourdomain.com`).
4. Azure provides a CNAME or TXT record to add at your DNS registrar.
5. Add the record, click **Validate** in Azure.
6. Azure provisions a free TLS certificate once validation passes.

DNS propagation: 5-30 minutes, occasionally up to 24 hours.

**Important:** Set up the custom domain before your first real use. localStorage is scoped to the origin URL, so changing the URL later means notes and progress from the old URL will not transfer automatically.

## Notes on notes вЂ” why no Google Docs / OneNote OAuth

Direct integration with Google Docs or OneNote requires:

- **A backend** to hold client secrets (which cannot live in a static PWA)
- **OAuth token management** вЂ” tokens expire, need refresh, fail offline
- **Azure Functions** or similar infrastructure, adding cost and complexity
- Breaking the app's offline guarantee whenever the OAuth flow needs a network call

The Journal tab uses the same approach as Anki: **write locally, export when ready**. The **Export .md** button downloads a single markdown file containing all your entries. You can paste that directly into Google Docs or OneNote, or open it in any text editor.

If you later want full OAuth integration, the architecture would be:

```
React app в†’ Azure Function (holds client secret) в†’ Google/Microsoft API
```

This is documented but not implemented. The static PWA approach is intentional.

## PWA notes

### Anki deck downloads

The `.apkg` files are static assets served from `dist/anki/`. They download directly from the browser вЂ” no server-side processing needed. On iOS, tapping a downloaded `.apkg` file opens it directly in the AnkiMobile app if installed.

### Journal data

Journal entries are stored in `localStorage` under the key `italian-bible-journal`. Each entry is a JSON object with `text` and `updatedAt` fields. The export function reads all entries and writes them to a single `.md` file.

### Progress data

Week completion is stored under `italian-bible-progress`. Format: `{ "1": true, "3": true, ... }`.

### Service worker behaviour

`registerType: 'autoUpdate'` in `vite.config.js` means the service worker updates silently in the background. Users get the new version on their next page load with no action required.

The `.apkg` files in `public/anki/` are precached by Workbox along with the app shell, so they are available offline after the first load.

### Installing to home screen

- **Android (Chrome):** Install prompt fires automatically, or tap the three-dot menu and choose **Add to Home Screen**. The in-app **Add to Home Screen** button appears when the browser fires the `beforeinstallprompt` event.
- **iOS (Safari):** Tap the **Share** button, then **Add to Home Screen**. The in-app button does not appear on iOS Safari вЂ” browser limitation, not a bug.

## Troubleshooting

**`npm error ERESOLVE` on install**

`vite-plugin-pwa` does not yet support Vite 8. `package.json` pins Vite to `^6.3.3`. Delete lockfile and reinstall:

```bash
rm -rf node_modules package-lock.json
npm install
```

**Anki decks show 0 cards after import**

This usually means Anki already has a deck with the same name. In Anki, go to **Browse**, find the deck, and delete the existing cards before importing again. Or rename the deck in `generate-anki.cjs` before regenerating.

**`.apkg` files download but won't open on iOS**

AnkiMobile must be installed. Tap the downloaded file in Safari's Downloads and choose **Open in AnkiMobile**. If that option does not appear, open AnkiMobile first, then tap the file from the Files app.

**Journal export is empty**

Only weeks with content are included in the export. Expand a week in the Journal tab, write something, and wait for the "Saved" indicator before exporting.

**App deploys but shows a blank page on Azure**

Check that **Output location** in the GitHub Actions workflow file is `dist`. Open `.github/workflows/azure-static-web-apps-*.yml` and look for `output_location: "dist"`.

**Anki files not found (404) after deploy**

Confirm `staticwebapp.config.json` is in `public/` and has been committed. The `/anki/*` exclusion in `navigationFallback` is required so `.apkg` requests are not rewritten to `index.html`.

**Offline mode not working after first deploy**

Load the app once while online to let the service worker cache all assets. Check **Application > Service Workers** in DevTools to confirm it is active. Then toggle offline.

**Actions workflow fails on first run (file not found)**

Azure commits the workflow file directly to the remote. Pull before you push again:

```bash
git pull origin main
git push
```
